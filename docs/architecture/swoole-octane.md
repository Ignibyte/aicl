# Swoole / Octane Infrastructure

## Purpose

Laravel Octane with Swoole replaces the traditional PHP-FPM request lifecycle with a long-lived process server. Instead of booting a fresh PHP process per request (FPM), Swoole starts a fixed pool of worker processes that persist across requests. The Laravel application boots once per worker, then handles thousands of requests without re-bootstrapping. This eliminates per-request framework boot overhead (typically 20-50ms) and enables coroutine-based concurrency, shared memory caches, and native WebSocket support.

Laravel Octane is the abstraction layer that integrates Swoole into Laravel's lifecycle. It handles:
- Resetting application state between requests (config, router, resolved singletons)
- Managing worker boot/shutdown lifecycle events
- Providing shared memory tables for cross-worker data
- Coordinating graceful restarts on code changes

The critical consequence of this architecture: **workers persist across requests**. Any state stored in static properties, singletons, or resolved services will leak between requests unless explicitly reset. This is the single most common source of bugs in Octane applications.

## Dependencies

- **Swoole 6.0.0** -- PHP extension compiled from source via DDEV web-build (`pecl install swoole-6.0.0`)
- **Laravel Octane v2.13** -- `laravel/octane` Composer package
- **Redis 7** -- Timer persistence, session storage, queue backend
- **nginx** -- TLS termination, static asset serving, reverse proxy to Swoole
- **supervisord** -- Process management within the DDEV container
- **DDEV v1.24.10** -- Development environment orchestration

Build file: `.ddev/web-build/Dockerfile.swoole` installs Swoole with sockets, OpenSSL, HTTP/2, mysqlnd, curl, and JSON support enabled.

## Key Decisions

### 1. nginx reverse proxy, not direct Swoole access

All traffic routes through nginx (`https://aicl.ddev.site`), which terminates TLS via the DDEV router, serves static assets from disk, and proxies dynamic requests to Swoole on `127.0.0.1:8000`. Direct Swoole access (`https://aicl.ddev.site:8443`) still works but is not the intended path -- it bypasses nginx's static asset serving and TLS handling.

### 2. Static asset fallback to Swoole

The nginx static asset location block uses `try_files $uri @octane` instead of `try_files $uri =404`. This is critical because dynamic routes like `/livewire/livewire.js` do not exist on disk -- they are generated by Laravel at runtime. Using `=404` would break Livewire, Filament assets, and any other dynamically-served resource that matches the static file extension regex.

### 3. No SSE -- WebSockets only

Server-Sent Events (SSE) were removed in Sprint H because each SSE connection holds an entire Swoole worker for the duration of the stream. With 4 workers, 4 concurrent SSE connections would exhaust the worker pool and block all other requests. AI streaming and real-time updates use Laravel Reverb (WebSockets) instead.

### 4. SwooleCache as L1 with graceful degradation

All Swoole-specific features (SwooleCache, SwooleTimer, Concurrent) degrade silently when Swoole is unavailable. `SwooleCache::get()` returns `null` on miss (triggers fallback to Redis/DB). `SwooleTimer::every()` persists to Redis but returns `false`. `Concurrent::run()` executes sequentially. This means the same codebase runs correctly under Octane, queue workers, artisan commands, and PHPUnit tests.

### 5. Worker 0 coordination for timers

Only worker 0 restores SwooleTimers on boot. Without this, all N workers would each register the same recurring timers, resulting in N duplicate job dispatches per interval.

### 6. Drupal-style config -- no .env for Octane

Octane configuration lives in `config/octane.php` with `env()` calls for stock Laravel compatibility. DDEV sets environment variables in the container. There is no `.env` file. AICL-specific config uses plain PHP values in `config/aicl.php` and `config/local.php`.

## Architecture

### Process Model

```
DDEV Container
  |
  +-- supervisord
  |     |
  |     +-- [webextradaemons group]
  |     |     +-- octane (Swoole server, port 8000)
  |     |     +-- reverb (WebSocket server, port 8080)
  |     |     +-- horizon (Queue manager)
  |     |     +-- schedule (Scheduler)
  |     |
  |     +-- nginx (ports 80/443)
  |
  +-- DDEV Router (external TLS termination)
```

Swoole spawns worker processes that each hold a copy of the booted Laravel application:

```
Swoole Master Process
  |
  +-- Manager Process
  |     |
  |     +-- Worker 0 (handles HTTP requests + restores timers)
  |     +-- Worker 1 (handles HTTP requests)
  |     +-- Worker 2 (handles HTTP requests)
  |     +-- Worker 3 (handles HTTP requests)
  |     +-- Task Worker 0 (async tasks)
  |     +-- Task Worker 1 (async tasks)
```

Workers share Swoole Tables (shared memory) but have independent PHP memory spaces for everything else.

### Request Flow

```
Client (HTTPS)
  |
  v
DDEV Router (TLS termination)
  |
  v
nginx (port 443, SSL cert)
  |
  +-- Static file exists on disk? --> Serve directly (30d cache, immutable)
  |
  +-- Static file extension but no file? --> @octane fallback (e.g., /livewire/livewire.js)
  |
  +-- Dynamic route --> proxy_pass http://127.0.0.1:8000
  |     |
  |     +-- Headers: Host, X-Real-IP, X-Forwarded-For, X-Forwarded-Proto, X-Forwarded-Port
  |     +-- WebSocket: Upgrade + Connection headers
  |     +-- Buffering: off (streaming support)
  |     +-- Timeouts: connect 60s, send 60s, read 300s
  |
  v
Swoole Worker (long-lived PHP process)
  |
  +-- Octane resets state (config, router, bindings)
  +-- AICL flushes per-request caches (permissions, badges, health)
  +-- Request handled by Laravel
  +-- Response returned to nginx
  +-- Octane: FlushOnce, FlushTemporaryContainerInstances
```

## DDEV Configuration

### Supervisor Config

**File:** `.ddev/.webimageBuild/octane.conf`

```ini
[program:octane]
group=webextradaemons
command=bash -c "php /var/www/html/artisan octane:start --server=swoole --host=0.0.0.0 --port=8000 --workers=4 --task-workers=2 --max-requests=500; ..."
directory=/var/www/html
autostart=false
autorestart=true
startsecs=3
startretries=15
stopasgroup=true
```

Key settings:
- **workers=4** -- HTTP request handler processes
- **task-workers=2** -- Async task handler processes (used by `Octane::concurrently()`)
- **max-requests=500** -- Worker restarts after 500 requests to prevent memory leaks
- **autostart=false** -- Managed by the `webextradaemons` supervisor group, not individually
- **autorestart=true** -- Supervisor restarts crashed workers automatically
- **startsecs=3** -- Must stay up 3 seconds to be considered "started" (matches the sleep-on-failure in the command)
- **startretries=15** -- Generous retries for database/service readiness on container boot
- **stopasgroup=true** -- Kills all child processes (workers) when stopping

The `webextradaemons` group (`.ddev/.webimageBuild/webextradaemons.conf`) bundles octane, reverb, horizon, and schedule together.

## Nginx Proxy Configuration

**File:** `.ddev/nginx_full/nginx-site.conf`

### TLS / Forwarded Headers

nginx determines `X-Forwarded-Proto` and `X-Forwarded-Port` via a `map` on the `$host` header. Requests to `*.ddev.site` are assumed HTTPS (the DDEV router terminates TLS). Internal requests (e.g., Selenium via `http://web`) stay HTTP.

```nginx
map $host $forwarded_proto {
    ~ddev\.site$    https;
    default         $scheme;
}
```

### Static Assets

```nginx
location ~* \.(css|js|ico|png|jpg|jpeg|gif|svg|webp|avif|woff2?|ttf|eot|map)$ {
    try_files $uri @octane;
    expires 30d;
    access_log off;
    add_header Cache-Control "public, immutable";
}
```

Matches common static file extensions. If the file exists on disk (`public/build/assets/app-Bx1a2b.js`), nginx serves it directly with 30-day cache headers. If the file does not exist (e.g., `/livewire/livewire.js`), the request falls through to the `@octane` named location, which proxies to Swoole.

**CRITICAL:** This MUST use `try_files $uri @octane`, NOT `try_files $uri =404`. Dynamic asset routes served by Laravel do not exist on disk.

### Dynamic Routes

```nginx
location / {
    proxy_pass http://octane;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_buffering off;
    proxy_cache off;
    proxy_connect_timeout 60s;
    proxy_send_timeout 60s;
    proxy_read_timeout 300s;
}
```

- **proxy_buffering off** -- Required for streaming responses (SSE was removed, but WebSocket upgrade and chunked responses still benefit)
- **proxy_read_timeout 300s** -- 5 minutes for long-running operations (AI processing, large exports)
- **Upgrade/Connection headers** -- Required for WebSocket connections (Livewire, Reverb)

### Listeners

```php
'listeners' => [
    WorkerStarting::class => [
        EnsureUploadedFilesAreValid::class,
        EnsureUploadedFilesCanBeMoved::class,
    ],
    RequestReceived::class => [
        ...Octane::prepareApplicationForNextOperation(),
        ...Octane::prepareApplicationForNextRequest(),
    ],
    OperationTerminated::class => [
        FlushOnce::class,
        FlushTemporaryContainerInstances::class,
    ],
    WorkerErrorOccurred::class => [
        ReportException::class,
        StopWorkerIfNecessary::class,
    ],
    WorkerStopping::class => [
        CloseMonologHandlers::class,
    ],
],
```

AICL registers additional listeners in `AiclServiceProvider::boot()`:
- `WorkerStarting` -> `WarmSwooleCaches` (populates SwooleCache tables with pre-loaded data)
- `WorkerStarting` -> `RestoreSwooleTimers` (worker 0 only; re-registers recurring/one-shot timers from Redis)

### Swoole Cache Table

```php
'cache' => [
    'rows' => 1000,
    'bytes' => 10000,
],
```

This is Octane's built-in cache store (`Cache::store('octane')`). AICL's `SwooleCache` is separate and uses its own registered tables.

### File Watching

```php
'watch' => [
    'app', 'bootstrap', 'config/**/*.php', 'database/**/*.php',
    'public/**/*.php', 'resources/**/*.php', 'routes',
    'composer.lock', 'config/local.php',
],
```

Used with `--watch` flag for automatic worker reload during development. Note `config/local.php` is explicitly included since it is gitignored and would not be caught by the `config/**/*.php` glob.

## State Management

This is the most critical section for diagnosing bugs in Octane applications.

### The Problem

Swoole workers are long-lived PHP processes. Unlike FPM (where each request gets a fresh process), Swoole workers persist:

- **Static properties** retain values across requests
- **Singletons** resolved from the container persist
- **Class-level state** in services, repositories, and helpers persists
- **Memoized values** (e.g., `private static ?string $cached = null`) persist

If request A sets a static property and request B reads it, B sees A's data. This causes user data leakage, stale cache values, and permission escalation bugs.

### How Octane Resets State

On each `RequestReceived` event, Octane calls `prepareApplicationForNextOperation()` and `prepareApplicationForNextRequest()`, which:
- Clone the application container
- Reset the router, translator, and URL generator
- Flush any bindings registered for flushing
- Reset request/session state

On `OperationTerminated`:
- `FlushOnce` clears `once()` memoization
- `FlushTemporaryContainerInstances` removes request-scoped bindings

### How AICL Manages State

AICL registers three cache managers that use Swoole's shared memory tables with event-driven invalidation:

| Manager | Table | Rows | TTL | Invalidation |
|---------|-------|------|-----|--------------|
| `PermissionCacheManager` | `permissions` | 2,000 | 300s | Spatie role/permission events + Eloquent model events |
| `NotificationBadgeCacheManager` | `notification_badges` | 1,000 | 60s | `DatabaseNotification` created/updated/deleted |
| `ServiceHealthCacheManager` | `service_health` | 10 | 30s | TTL-only (external services) |

Registration in `AiclServiceProvider::boot()`:

```php
Swoole\Cache\PermissionCacheManager::register();
Swoole\Cache\NotificationBadgeCacheManager::register();
Swoole\Cache\ServiceHealthCacheManager::register();

Event::listen(WorkerStarting::class, WarmSwooleCaches::class);
Event::listen(WorkerStarting::class, RestoreSwooleTimers::class);
```

### Three-Tier Cache Hierarchy

```
L0: Per-instance property (request scope, dies with request)
 |
 v miss
L1: SwooleCache / Swoole Tables (cross-worker shared memory, sub-microsecond)
 |
 v miss
L2: Redis / PostgreSQL / HTTP (network I/O)
```

- **L0** -- PHP property on the service instance (e.g., `$this->esAvailable`). Lives for one request.
- **L1** -- SwooleCache. Shared memory across all workers. JSON-serialized values with lazy TTL expiration and event-driven invalidation.
- **L2** -- Source of truth. PostgreSQL for data, Redis for sessions/queues, HTTP for external service health.

### SwooleTimer Persistence

Default timers registered in `AiclServiceProvider::boot()`:

| Timer Key | Interval | Job | Purpose |
|-----------|----------|-----|---------|
| `health-refresh` | 300s (5 min) | `RefreshHealthChecksJob` | Background health check refresh |
| `delivery-cleanup` | 3600s (1 hr) | `CleanStaleDeliveriesJob` | Clean up stale notification deliveries |

Timer definitions persist in Redis at `aicl:timers:{key}`. On worker restart, `RestoreSwooleTimers` (listening on `WorkerStarting`) calls `SwooleTimer::restore()` on worker 0 only, re-registering all persisted timers.

### Rules for Writing Octane-Safe Code

1. **Never use static properties for per-request state.** Use the container, request attributes, or `once()`.
2. **Never store user-specific data in class properties that outlive the request.** Use `SwooleCache` with per-user keys and TTL, or rely on the session.
3. **Flush singletons that accumulate state.** Register them in `config('octane.flush')` or use `FlushTemporaryContainerInstances`.
4. **Timer callbacks must dispatch jobs, never run business logic inline.** Timer callbacks execute in the Swoole event loop -- blocking one blocks all workers.
5. **Test with `ddev octane-reload` after code changes.** Octane workers cache the compiled application. Without reload, you are running stale code.

## Concurrency Utilities

AICL provides coroutine-based concurrency primitives that leverage Swoole's cooperative multitasking:

### Concurrent

**File:** `packages/aicl/src/Swoole/Concurrent.php`

```php
use Aicl\Swoole\Concurrent;

// Parallel execution with keyed results
$results = Concurrent::run([
    'users' => fn () => Http::get('/api/users')->json(),
    'orders' => fn () => Http::get('/api/orders')->json(),
], timeout: 5.0);

// Fan-out with concurrency limit
$results = Concurrent::map($records, fn ($r) => process($r), concurrency: 5);

// First successful result wins
$result = Concurrent::race([
    'fast' => fn () => FastProvider::query($q),
    'slow' => fn () => SlowProvider::query($q),
]);
```

Falls back to sequential execution when `Coroutine::getCid() <= 0` (not in Swoole context).

### SwooleCache

**File:** `packages/aicl/src/Swoole/SwooleCache.php`

Cross-worker shared memory cache. Sub-microsecond reads. JSON serialization. Lazy TTL expiration. Event-driven invalidation.

```php
use Aicl\Swoole\SwooleCache;

// Register at boot time (service provider)
SwooleCache::register('my_cache', rows: 500, ttl: 60, valueSize: 2000);

// Read-through pattern
$value = SwooleCache::get('my_cache', $key);
if ($value === null) {
    $value = expensive_computation();
    SwooleCache::set('my_cache', $key, $value);
}

// Event-driven invalidation
SwooleCache::invalidateOn('my_cache', ModelUpdated::class,
    fn ($event) => "key:{$event->model->id}");
```

Returns `null`/`false`/`0` when Swoole is unavailable -- callers naturally fall through to the next cache tier.

### SwooleTimer

**File:** `packages/aicl/src/Swoole/SwooleTimer.php`

Named timers with Redis persistence and job dispatch.

```php
use Aicl\Swoole\SwooleTimer;

// Recurring -- dispatch job every 5 minutes
SwooleTimer::every('cleanup', 300, CleanupJob::class);

// One-shot -- dispatch job after 10 minutes
SwooleTimer::after('reminder-42', 600, ReminderJob::class, [42]);

// Cancel
SwooleTimer::cancel('reminder-42');
```

Persists to Redis regardless of Swoole availability. Timer restoration on worker boot is handled automatically.

## Common Issues & Troubleshooting

### BACKOFF after ddev restart

**Symptom:** Octane enters BACKOFF state after `ddev restart`. Supervisor logs show repeated start failures.

**Cause:** Stale PID file from the previous container lifecycle. Swoole refuses to start because it thinks another instance is running.

**Fix:**
```bash
ddev exec artisan octane:stop
ddev exec supervisorctl restart webextradaemons:octane
```

### State leaks between requests

**Symptom:** User A sees User B's data. Permissions are wrong. Cached values from previous requests appear.

**Cause:** Static properties, singletons, or memoized values not reset between requests.

**Diagnosis:**
1. Search for `private static` and `protected static` in the affected code
2. Check if the service is registered as a singleton in the container
3. Look for class-level property assignments that should be request-scoped

**Fix:** Use `SwooleCache` with per-user keys, register the binding in `config('octane.flush')`, or use request-scoped storage.

### Filament Js::make() not updating

**Symptom:** JavaScript changes via `Js::make()` in Filament components do not appear after `ddev octane-reload`.

**Cause:** Filament's JS assets are compiled and cached separately from Octane's worker state. Reloading workers does not republish assets.

**Fix:**
```bash
ddev exec artisan filament:assets
```

### url() / route() generate wrong port

**Symptom:** `url()` and `route()` generate URLs with `:443` but the Octane direct port is `:8443`. AJAX calls fail with connection refused.

**Cause:** Octane's `https` config generates URLs with the standard HTTPS port (443). When accessing Swoole directly on 8443, the port mismatch breaks same-origin requests.

**Fix:** Use relative paths for same-origin AJAX (`/api/endpoint` instead of `url('/api/endpoint')`). Or, use the nginx proxy (`https://aicl.ddev.site` without port) where the port is correct.

### Worker not picking up code changes

**Symptom:** Code changes have no effect. Old behavior persists.

**Cause:** Swoole workers cache the compiled application. They do not re-read PHP files on each request.

**Fix:**
```bash
ddev octane-reload
```

For config changes in `config/local.php`, also run:
```bash
ddev exec artisan config:clear
ddev octane-reload
```

### Memory issues / workers restarting unexpectedly

**Symptom:** Workers restart frequently. Logs show memory-related warnings.

**Cause:** Memory leak in application code. Workers accumulate memory across requests.

**Explanation:** Two safety mechanisms prevent runaway memory:
1. **max-requests=500** -- Workers automatically restart after 500 requests
2. **garbage=50** -- Workers trigger GC when memory exceeds 50MB

If workers are restarting too frequently, the application is likely leaking memory (growing arrays in static properties, unclosed resources, accumulating event listeners).

**Diagnosis:** Check `ddev exec supervisorctl status webextradaemons:octane` for restart frequency. Look for static array properties that grow unbounded.

### SSE connections exhaust workers

**Symptom:** After a few concurrent SSE connections, the site becomes unresponsive.

**Cause:** Each SSE connection holds an entire Swoole worker for the duration of the stream. With 4 workers, 4 SSE connections = 0 workers available for other requests.

**Fix:** SSE was removed from AICL in Sprint H. Use Reverb/WebSockets instead. If you need streaming, dispatch to a queue job that broadcasts progress via Reverb.

### Redis cache returns integers as strings

**Symptom:** `===` comparisons fail for integer values retrieved from Redis cache.

**Cause:** Redis returns all values as strings, even integers.

**Fix:** Cast explicitly: `(int) Cache::get('key')` for `===` comparisons. This affects Swoole timer data and any Redis-cached values used in strict comparisons.

## Production Considerations

### Worker Scaling

- **Workers:** Set to 2x CPU cores for CPU-bound workloads, 4x for I/O-bound (default: `--workers=4`)
- **Task workers:** Increase for heavy async operations like bulk exports or AI processing (default: `--task-workers=2`)
- **Max requests:** Prevent memory leaks by restarting workers periodically (default: `--max-requests=500`). Lower if memory pressure is observed; raise if restarts cause latency spikes.

### Memory Budget

- Each worker consumes the full Laravel boot memory (~40-80MB) plus per-request allocations
- SwooleCache tables are shared memory (allocated once, visible to all workers)
- Current table budget: ~10.4 MB shared across all workers
- GC threshold at 50MB per worker catches leaks before OOM

### Static Assets

In production, use nginx (or a CDN) for all static assets. This offloads Swoole workers entirely from serving CSS, JS, images, and fonts. The nginx config in this project is a good production starting point -- just add CDN headers or upstream caching as needed.

### TLS Termination

TLS should be terminated by the load balancer or nginx, never by Swoole directly. Swoole's `https` config option tells Laravel to generate `https://` URLs, but the actual TLS handshake happens upstream.

### Health Monitoring

Monitor Swoole worker status via supervisor:
```bash
supervisorctl status webextradaemons:octane
```

Monitor application-level health via the health check registry (SwooleCheck, PostgresCheck, RedisCheck, etc.), which is cached in `ServiceHealthCacheManager` with a 30-second TTL.
